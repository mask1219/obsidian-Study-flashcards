# PRD-完善AI模式多模型配置

## 1. Summary

本 PRD 用于定义 Obsidian 学习闪卡插件中“生成模式”里的 AI 模式升级方案。目标是让用户可以保存多个模型配置，为每个模型单独设置调用参数，并在设置页中选择当前生效模型来生成知识卡片。

本次需求还包括对现有设置页的样式调整。新的 AI 配置区需要以一个可折叠的大区域展示，展开后可以看到模型列表和对应操作。

## 3. Background

### 3.1 当前情况

当前插件已经支持以下能力：

- 生成模式支持 `rule`、`ai`、`hybrid`
- AI 模式可以调用模型生成知识卡片
- 设置页中已有单套 AI 配置
- 当前字段包括 Provider、API 地址、API Key、模型名、附加提示词、连接测试

### 3.2 当前问题

现有 AI 模式只有一套全局配置，带来以下问题：

- 用户无法同时保存多组模型参数
- 用户切换模型时需要反复手动改写配置
- 不同模型之间无法做独立管理
- 设置页 AI 区域较平铺，不利于后续扩展
- 旧结构难以支撑“配置列表 + 当前生效模型”的使用方式

### 3.3 为什么现在做

插件已经具备 AI 生成知识卡片的基础能力。现在继续完善多模型配置，可以让 AI 模式真正可用、可切换、可管理，也为后续扩展更多模型来源打下基础。

## 4. Objective

### 4.1 目标

让用户在插件内维护多个 AI 模型配置，并为每个模型单独设置调用参数。用户可以在设置页直接选择一个“当前生效模型”，之后生成知识卡片时直接使用该模型配置。

### 4.2 业务和用户价值

对用户的价值：

- 减少反复改配置的成本
- 支持同时使用多个模型或多个服务商
- 更清楚地管理模型用途和参数
- 出错时可以快速看懂问题位置

对产品的价值：

- 提升 AI 模式可用性
- 提升设置页扩展性
- 为后续接入更多模型能力提供基础结构

### 4.3 成功指标

KR1：用户可以新增、编辑、删除、复制、排序多个模型配置，并能成功保存。

KR2：用户可以在设置页明确选择一个当前生效模型，生成知识卡片时使用该模型配置。

KR3：当 AI 调用失败时，系统用中文明确提示出错点，不回退到规则生成。

KR4：旧版单模型字段不再作为正式配置来源，新版本使用新的多模型配置结构。

KR5：AI 模型配置区可以折叠和展开，展开后展示模型列表及操作入口。

## 5. Market Segment(s)

本功能主要面向以下用户：

- 在 Obsidian 中进行系统学习和复习的用户
- 需要用不同模型生成学习卡片的用户
- 会使用多个 API 服务商或多个模型版本的用户
- 希望长期维护个人知识卡片流程的重度用户

这些用户的共同问题不是“能不能生成卡片”，而是“能不能稳定、方便地切换和管理不同模型配置”。

## 6. Value Proposition(s)

### 6.1 用户任务

用户希望完成这些任务：

- 保存多个模型配置，避免每次重新填写
- 快速切换当前用于生成卡片的模型
- 明确知道当前到底在使用哪一个模型
- 出错时能看懂原因并自己修正

### 6.2 用户收益

- 配置一次，多次复用
- 模型切换更快
- 设置页更清晰
- 错误提示更友好

### 6.3 相比当前版本的改进

当前版本只能维护单套 AI 参数。本次升级后，插件会支持“模型配置列表 + 当前生效模型”的使用方式，明显降低使用门槛，也更适合真实场景。

## 7. Solution

### 7.1 UX / 信息结构

设置页中的 AI 相关内容调整为一个可折叠的大区域。

交互要求如下：

- 默认展示 AI 配置区域标题
- AI 配置区域默认折叠
- 用户可以展开或折叠整个 AI 模型配置区域
- 展开后展示模型列表
- 列表中每条模型配置提供明确的操作入口
- 页面中提供“当前生效模型”选择能力

不要求在生成闪卡时额外弹窗选模型。生成时直接使用设置页中已选定的当前生效模型。

### 7.2 数据结构

新增“模型配置”概念。每条模型配置包含以下字段：

- 配置名称
- Provider
- API URL / Base URL
- API Key
- 模型名
- 附加提示词

全局新增字段：

- 当前生效模型 ID
- 模型配置列表

以下旧字段直接废弃，不做迁移：

- `aiProvider`
- `aiApiUrl`
- `aiApiKey`
- `aiModel`
- `aiPrompt`

说明：

- 本次不做旧数据自动迁移
- 升级后以新的多模型配置结构为准
- 若用户未新建任何模型配置，则 AI 模式不可用，并需要明确提示

### 7.3 功能需求

#### 7.3.1 模型配置管理

系统必须支持以下操作：

- 新增模型配置
- 编辑模型配置
- 删除模型配置
- 复制模型配置
- 设为默认
- 排序

补充说明：

- “设为默认”本质上应更新“当前生效模型”
- 删除当前生效模型时，系统不自动切换到其他模型
- 删除当前生效模型后，系统直接清空“当前生效模型”
- 若删除后仍有其他模型，系统需要用中文提示用户重新选择当前生效模型
- 若列表为空，AI 模式和混合模式都不可正常调用 AI
- 复制模型配置后，默认命名规则为 `原配置名称-副本`
- 若 `原配置名称-副本` 已存在，则继续追加序号，例如 `原配置名称-副本2`

#### 7.3.2 当前生效模型

设置页中必须有明确入口，让用户选择一个当前生效模型。

行为要求：

- 生成知识卡片时直接读取当前生效模型
- 不在生成动作中再次询问用户
- 当前生效模型必须是配置列表中的某一项

#### 7.3.3 生成流程

当生成模式为 `ai` 或 `hybrid` 时：

- 系统读取当前生效模型
- 按该模型的 Provider 和参数进行调用
- 成功后返回知识卡片结果

当配置缺失时：

- 系统阻止调用
- 用中文明确提示缺少哪一项

当用户保存模型配置时：

- 系统需要做基础必填校验
- 至少校验配置名称、Provider、API URL、API Key、模型名
- 未通过校验时，不允许保存
- 提示必须使用中文，并明确指出缺失字段

#### 7.3.4 错误处理

错误提示必须使用中文，并尽量指出具体出错点。

至少覆盖以下场景：

- 未选择当前生效模型
- 模型配置为空
- 缺少 API URL
- 缺少 API Key
- 缺少模型名
- Provider 不支持
- 网络请求失败
- 接口返回非预期结构
- JSON 解析失败
- 鉴权失败
- 限流或配额不足

错误文案目标：

- 用户能看懂
- 用户知道出错在“配置”“网络”“接口返回”还是“鉴权”
- 用户知道下一步应该修改哪里

#### 7.3.5 失败策略

本次需求明确如下：

- `ai` 模式调用失败时，直接中文报错，不回退
- `hybrid` 模式调用失败时，直接中文报错，不回退

这意味着 `hybrid` 模式在本次需求下不再承担“失败自动回退到规则生成”的职责。

#### 7.3.6 连接测试

连接测试需要适配新的多模型配置结构。

要求如下：

- 测试只基于当前正在编辑的模型配置
- 测试结果用中文提示
- 失败时尽量给出具体原因

### 7.4 样式要求

设置页样式要围绕“AI 模型配置区域”做整理，重点不是视觉装饰，而是结构清晰。

要求如下：

- AI 模型配置是一个独立大区域
- 区域支持折叠和展开
- 区域默认折叠
- 展开后模型列表层次清晰
- 当前生效模型状态需要明显可见
- 每条配置的操作按钮应便于理解
- 配置编辑区域不要与其他插件设置混在一起

### 7.5 技术要求

- 调整插件设置数据结构，支持多模型配置
- 更新设置页渲染逻辑
- 更新 AI 调用入口，从旧单配置读取改为从当前生效模型读取
- 更新连接测试逻辑
- 增加保存时必填校验逻辑
- 更新相关校验逻辑和错误提示
- 清理或废弃旧单模型字段的使用路径

### 7.6 假设

- 用户愿意在设置页完成模型管理，不需要在生成时二次选择模型
- 一个“当前生效模型”足以满足主要使用场景
- 当前已有的 Provider 体系仍可继续复用到多模型配置结构中

## 8. Release

### 8.1 第一版范围

第一版必须包含：

- 新的多模型配置数据结构
- 当前生效模型选择
- 模型配置的新增、编辑、删除、复制、设为默认、排序
- AI 配置区折叠展示
- 中文错误提示
- 新连接测试逻辑
- 旧字段废弃

### 8.2 后续版本可考虑

- 导入导出模型配置
- API Key 加密存储
- 按不同生成场景绑定不同模型
- 为不同 Provider 提供更细的专属参数
- 模型配置搜索和筛选

### 8.3 风险与注意事项

- 旧字段直接废弃，会让旧用户需要重新配置模型
- 多模型结构会提升设置页复杂度，需要控制交互负担
- 不回退到规则生成后，AI 失败会更直接暴露问题，需要保证报错足够清楚
- 排序、复制、删除、设为默认这些操作之间要避免状态错乱
